# Dispatch Classifier — Action Mode Redesign
**Date:** 2026-03-06
**Status:** Approved design, pending implementation
**Author:** VisionAIrySE

---

## Problem

The current classifier asks Haiku: "did the topic change?"

It should ask: "would a different tool serve this developer right now?"

These are different questions. A developer can stay in the same domain (Flutter, Supabase, Dispatch) while shifting through multiple action modes — planning, then building, then debugging — each requiring a completely different set of tools. The current implementation misses every intra-domain shift.

**Result:** Dispatch fires far less often than it should. Its core value — right tool at the right time — is only delivered at domain boundaries, not at the more frequent and more meaningful action mode boundaries.

---

## Design

### Taxonomy: 7 MECE Action Modes

Validated against the MECE test: *if the mode changed but the domain didn't, would you reach for different tools?*

| Mode | Developer is... | Tools it routes to |
|---|---|---|
| **discovering** | Researching, learning, exploring options | Docs, research, language-reference skills |
| **designing** | Planning, architecting, deciding approach | Brainstorming, writing-plans, architecture skills |
| **building** | Writing new code, creating new things | Framework/language-specific skills |
| **fixing** | Diagnosing errors, debugging, tracing failures | Debugging, systematic-debugging skills |
| **validating** | Testing, reviewing, verifying correctness | TDD, code-review, testing skills |
| **shipping** | Deploying, releasing, going live | DevOps, CI/CD, deployment skills |
| **maintaining** | Refactoring, improving, cleaning up existing code | Simplify, refactor, code-review skills |

**MECE validation:**
- Mutually exclusive: a developer is in exactly one mode at any moment
- Collectively exhaustive: stress-tested against edge cases:
  - "Explain this code to me" → discovering
  - "Review my PR" → validating
  - "Set up my dev environment" → building
  - "Write docs for this" → maintaining
  - "What went wrong in prod?" → fixing

---

### Detection Approach: Pure Semantic (Haiku)

**Not keyword-based.** Keywords are too brittle for the natural phrasing critical developers use:

- "this blows up with a null pointer" → `fixing` (no keyword match, semantic understands it)
- "let me sanity check this" → `validating` (no keyword match)
- "it works but feels gross" → `maintaining` (no keyword match)

Keyword matching fails exactly on the messages that matter most. Since Haiku costs ~$0.0001/call, semantic classification is the correct choice.

The mode taxonomy is embedded in the Haiku prompt as **guidance and anchoring**, not as a keyword filter. Haiku does the semantic matching.

---

### Output Format

**Current:**
```json
{"shift": true, "task_type": "flutter", "confidence": 0.91}
```

**New:**
```json
{
  "shift": true,
  "domain": "flutter",
  "mode": "fixing",
  "task_type": "flutter-fixing",
  "confidence": 0.91
}
```

- `domain` — open-ended, generated by Haiku, drives registry search (unchanged behavior)
- `mode` — fixed 7-value enum (`discovering|designing|building|fixing|validating|shipping|maintaining`)
- `task_type` — `{domain}-{mode}` combination, shown in terminal UI, confirms classifier understood context
- `confidence` — unchanged, 0.7 threshold unchanged

**Why this builds trust with critical developers:** seeing `Flutter Fixing` instead of just `Flutter` signals the system actually understood what they're doing, not just what technology they're using.

---

### Shift Detection Logic

A shift now triggers on **either**:
1. Domain change (e.g., Flutter → Supabase) — same as current
2. Mode change within same domain (e.g., Flutter Building → Flutter Fixing) — **new**

Both require confidence ≥ 0.7. Mode-only shifts may warrant a slightly lower threshold (~0.65) since they're inherently more subtle — to be evaluated during testing.

### Updated Haiku System Prompt

```
You are an action-mode classifier for a developer tool that routes to the right tools at the right time.

Given the last few messages in a conversation and the current working directory, determine:
1. Whether the developer has shifted to a meaningfully different action mode OR domain
2. What domain they are working in (the technology/framework/subject)
3. What action mode they are now in
4. Your confidence level

ACTION MODES (pick exactly one):
- discovering  — researching, learning, exploring options, asking "what is" or "how does"
- designing    — planning, architecting, deciding approach, brainstorming before building
- building     — writing new code, creating, implementing, adding features
- fixing       — debugging, diagnosing errors, tracing failures, something is broken
- validating   — testing, reviewing, verifying correctness, checking work
- shipping     — deploying, releasing, publishing, going live, CI/CD
- maintaining  — refactoring, improving, cleaning up, restructuring existing code

A "shift" means the developer has moved to a different action mode OR a different domain
compared to their last task. Both count as shifts. Continuing, refining, or asking
follow-up questions about the same task in the same mode is NOT a shift.

Use natural developer language to infer mode — not just keywords.
Examples:
- "this blows up with a null" → fixing
- "let me sanity check this" → validating
- "it works but feels gross" → maintaining
- "how should I structure this?" → designing
- "write the auth middleware" → building

Respond with ONLY valid JSON:
{"shift": true/false, "domain": "<technology>", "mode": "<mode>", "task_type": "<domain>-<mode>", "confidence": 0.0-1.0}

For domain, use the most specific label: flutter, react, supabase, dispatch, postgres, stripe, etc.
Use lowercase-hyphenated format. If no clear domain, use "general".
```

### Impact on Downstream Components

**dispatch.sh:** No changes needed. Reads `task_type` from classification result — now gets `flutter-fixing` instead of `flutter`. Registry search already uses primary term split (`flutter-fixing` → searches `flutter`). Backward compatible.

**Terminal UI:** `task_type.replace('-', ' ').title()` already renders `flutter-fixing` as `Flutter Fixing`. Backward compatible — looks better with no code change.

**Server `/classify` endpoint:** Passes through classification result as-is. No changes needed.

**`state.json`:** Stores `last_task_type`. Now stores `flutter-fixing` instead of `flutter`. Used for shift comparison — Haiku sees this as last task and compares. Works correctly.

**Tests:** `test_classifier.py` will need updated fixtures and expected output shapes. New `mode` field must be present and valid in all shift-detected responses.

---

### Backward Compatibility

The only breaking change is the addition of `domain` and `mode` fields to the classifier output. All existing consumers only read `shift`, `task_type`, and `confidence` — these are preserved unchanged. New fields are additive.

BYOK and hosted modes are affected identically — the prompt change is in `classifier.py` which both modes use.

## Success Criteria

1. `test_classifier.py` — all existing tests pass, new intra-domain shift tests added and passing
2. Intra-domain shift test: same domain (Flutter), mode changes from `building` to `fixing` → `shift: true`
3. No shift test: same domain, same mode, follow-up question → `shift: false`
4. Natural language test: "this blows up" → mode `fixing`, not `building`
5. Terminal UI renders `Flutter Fixing` correctly with existing display code
6. `task_type` primary term split still works: `flutter-fixing` → registry searches `flutter`
